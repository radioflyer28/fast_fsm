**Method: test_trigger_timing_condition_throughput** (marked `@pytest.mark.slow`)

Follow the exact pattern of `test_trigger_min_throughput`:

1. Create a minimal 2-state toggle FSM (`state_a` ↔ `state_b`) with `TimeoutCondition(999999.0)` as guard on both transitions (huge timeout so it never expires during the benchmark — always passes).
2. Warm up: 1000 iterations of `trigger("toggle")`.
3. `gc.collect()`
4. Time a batch of 200,000 `trigger("toggle")` calls using `time.perf_counter()`.
5. Calculate `ops_per_sec = iterations / elapsed`.
6. Detect compiled vs pure-Python mode using the same `importlib.util.find_spec("fast_fsm.core")` pattern from `test_trigger_min_throughput`.
7. Set floor: 200,000 if compiled, 30,000 if pure-Python.
8. Assert `ops_per_sec >= floor` with a descriptive error message matching the style of the existing test:
   ```
   f"trigger() throughput with timing condition guard {ops_per_sec:,.0f} ops/sec "
   f"is below the {'compiled' if compiled else 'pure-Python'} floor of "
   f"{floor:,} ops/sec. This indicates a performance regression.\n"
   f"  Elapsed for {iterations:,} transitions: {elapsed:.3f}s\n"
   f"  Mode: {'compiled (mypyc)' if compiled else 'pure Python'}"
   ```

**Docstring:** Explain this test verifies PERF-01 — that a single `time.monotonic()` call in the condition hot path does not degrade throughput below 200k ops/sec. The `TimeoutCondition(999999.0)` ensures the condition always passes so we're measuring condition overhead, not blocked transitions.
